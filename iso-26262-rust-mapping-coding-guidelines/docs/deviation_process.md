# Deviation Process

Use this process when a guideline cannot be met in a given change.

## Required inputs

- Guideline ID and enforcement mode
- Justification for deviation
- Impact and risk assessment
- Proposed mitigations
- Expiration/review date for the deviation

## Approval

- Deviation requires reviewer sign-off from the designated safety/code owner.
- Deviations touching `unsafe`, FFI, or concurrency-sensitive logic require elevated review.

## Evidence

- Link to code review/audit checklist
- Link to test/static analysis evidence
- Link to any follow-up issue tracking remediation

## Closure

- Close deviation when compliance is restored or rule is superseded/deprecated with rationale.
